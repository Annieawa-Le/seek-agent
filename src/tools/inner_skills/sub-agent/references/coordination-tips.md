# 主模型与子模型的协作技巧

## 委派策略

1. **明确任务边界** — 给子模型的任务描述越具体，结果越可靠。避免模糊的"看看这个"。

2. **选择合适的模式**
   - 子模型需要**理解对话背景** → clone
   - 子模型需要**专业角色** → mission
   - 需要**旁路监控** → listen

3. **工具权限控制** — 只给子模型完成任务所需的工具。不需要写文件的任务不要给 patch 工具。

## 查询与等待

- **快速状态查询**：`agent_query(name="xxx")` — 立即返回状态快照
- **等待子模型完成**：`agent_query(name="xxx", waitForCompletion=true)` — 挂起等待，适合需要结果才能继续的场景
- **向子模型提问**：`agent_query(name="xxx", question="你对这个方案怎么看？")` — 子模型的 LLM 会直接回答，不调工具

## 生命周期管理

- **不要保留不需要的子模型** — 任务完成后及时 `agent_fire` 销毁
- **同名创建会自动替换** — 先销毁旧的再创建新的
- **退出程序自动清理** — 所有子模型会在退出时被销毁

## 独立 Patch 暂存区（clone/mission）

clone 和 mission 模式的子模型拥有**独立的 patch 暂存区**，它们做的修改不会影响主模型的暂存区。

- `add_patch` / `del_patch` / `modify_patch` → 写入子模型的本地暂存区
- `ensure_patch(true)` → 直接修改文件（不需要主模型二次确认）
- `ensure_patch(false)` → 丢弃所有本地暂存
- `check_patch(index)` → 查看本地暂存区中某个 patch 的详情
- `revise_patch(index, ...)` → 替换本地暂存区中的某个 patch
- `a_submission` → 提交工作结果给主模型，报告中应包含修改的文件和内容摘要

## 不建议的场景

- ❌ **不要用子模型做实时对话** — 子模型的结果通过提交注入，不是实时流式的
- ❌ **不要让子模型操作主模型的核心状态** — 子模型不应调用 `memory_focus`、`memory_shorten` 等
- ❌ **不要创建递归子模型** — 子模型内部调用 `spawn_agent` 会导致不可预测的行为
